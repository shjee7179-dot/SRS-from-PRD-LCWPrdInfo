# SRS 버전 비교 검토 보고서 (v0.3 vs v1.0)

본 보고서는 기능적 개선 사항이 반영된 `SRS_v0.3.md` (개선본)와 초기 문서인 `SRS-001_VPS_v1.0_Opus.md` (원본) 문서를 비교하여 (1) 기술 스택 명확성, (2) MVP 목표 및 가치전달 조정 내용, (3) 기타 설계적 차이점 이라는 세 가지 관점에서 검토 및 정리한 문서입니다.

---

## 1. 기술 스택의 명확성 비교

개선된 v0.3 버전은 ‘Vibe Coding(AI를 활용한 빠른 개발)’과 ‘인프라 복잡도 축소’를 위해 모호했던 MSA 아키텍처를 단일 풀스택 프레임워크로 명확화하고 강제 항목(C-TEC)을 문서 전반에 추가했습니다.

| 구분 | 원본 문서 (v1.0) | 개선 문서 (v0.3) | 변경 사유 및 기대 효과 |
| :--- | :--- | :--- | :--- |
| **아키텍처 구조** | 마이크로서비스 아키텍처 (MSA)<br>- 10개의 복잡한 단위 서비스로 분리<br>- API Gateway 포함 | **Next.js 모노리식 아키텍처**<br>- 단일 통합 풀스택 프레임워크 | 인프라 유지비용 제거 및 백엔드 개발 복잡도 회피 (바이브코딩 최적화 목적) |
| **프론트/백엔드 연동** | REST API 기반 통신<br>- UI와 백엔드 통신 주체 불분명 | **Server Actions** 및 **Route Handlers**<br>- Client Component에서 서버 직접 호출 | 개발 속도 단축 및 API 작성·관리 공수 최소화 |
| **데이터베이스 및 ORM** | PostgreSQL 중앙통제 + Redis Cache | **Prisma ORM** + 개발용 로컬 SQLite + **Supabase (PostgreSQL)** 배포 통합 | Redis 등의 불필요한 인프라 종속성 제거 및 초기 설정 복잡성 완화 |
| **인증 및 권한 통제** | 커스텀 OAuth 2.0 서비스 (AuthSvc) 구축 | 자체 Session 구현 배제, **Supabase Auth** 및 **RLS(Row Level Security)** 강제 | 인증 구현 지연 요소를 원천 차단하고 데이터 계층 권한 통제로 보안성 확보 |
| **AI 기능 통합** | 별도의 AI Mapping Engine 서버 / API 연동 | 프레임워크 내장형 **Vercel AI SDK**, **Google Gemini API** 호출 | 언어모델 제어를 Next.js 내부 Action 단위로 통합하여 외부 응답 타임아웃/오류 방지 |
| **UI 및 컴포넌트** | 구체적인 프론트엔드/UI 명시 항목 부재 | **Tailwind CSS** 및 **shadcn/ui** 사용 강제 | 디자인 시스템 규격화로 AI 코드 생성 일관성 및 품질 확보 |
| **인프라 배포 관리** | 클라우드 인프라 분산 배포 | **Vercel** 서버리스 환경으로 단일 배포 | CI/CD Git Push 자동화로 데브옵스(DevOps) 소요시간 삭제 |

## 2. MVP 목표 및 가치전달 조정 내용

v0.3 단계에서는 'Phase 1 MVP 단계에서의 목표 달성 가능성'에 집중하여, 외부망 연동을 미루거나 보안 제약을 현실적으로 우회하는 등 기회 비즈니스 가치 전달에 꼭 필요한 범위 밖의 항목들을 조정했습니다.

| 구분 | 원본 문서 (v1.0) | 개선 문서 (v0.3) | 조정 목적 및 비즈니스적 가치 |
| :--- | :--- | :--- | :--- |
| **기존 LMS 시스템<br>LTI 연동 방안** | 외부 LMS와의 LTI 1.3 기반 연동 기능을 Phase 1에서 실 구축 (REQ-FUNC-035) | 1단계에서는 **Mock-up (외부 망 없는 자체 모의 테스트) 수준**까지만 구현하여 연기 | 외부 연동 디버깅 지옥 (타임아웃 이슈 등)을 사전에 차단하고 핵심 가치 구현에 리소스 집중 |
| **Open Badge 발행** | 외부 Open Badge Infra(OB) 서버와 직접 통신 및 외부 지갑형 배치 발급 | 직접 외부 발급을 막고 플랫폼 **자체 DB(Supabase)에 Assertion JSON 규격 격납** | 우선 내부 플랫폼 생태계에서 진위 확인(검증 URL)을 충족시켜 MVP 최소 동작 증명 |
| **AI 시스템의 사용성<br>및 매핑 전략** | 높은 정확도의 AI 자동 포맷 매핑 구현 중심 | 자동 매핑보다 **수동 보정 유연성(Human-in-the-loop)**에 초점 (REQ-FUNC-002) | 높은 AI 정확도 달성 노력에 따른 매몰 비용 우려를 줄이고, 관리자 보정 방식으로 빠른 기능 오픈 |
| **보안/CSAP 극복 전략** | "CSAP 미보유 SaaS 공공기관 반려율 100%" 제약 위주의 엄격한 기준 | **보안 요건 실용적 완화** 및 단일 테넌트 분리 구조 지향으로 방향 우회 | 공공 도입 장벽을 낮춰서 단기적 레퍼런스(계약 성사) 3건 확보라는 전략적 MVP 목표 달성 지원 |

## 3. 기타 차이점

다이어그램 및 구체적 명세 기술 측면에서 Vibe Coding과 Next.js 기반의 환경에 유리한 문서로 전체적인 내용이 구체화되었습니다. 

| 구분 | 원본 문서 (v1.0) | 개선 문서 (v0.3) | 내용 요약 |
| :--- | :--- | :--- | :--- |
| **시퀀스 다이어그램<br>관점 전환** | 마이크로서비스 간 통신 구조 표현 (Gateway → Auth → Service → DB 등 파편화된 흐름) | **Next.js 특화 시퀀스로 전환**<br>(Browser/Client Component → Server Action → Prisma/SDK) | 개발자가 바로 시퀀스를 코드로 치환할 수 있도록 실전적인 구조로 상세화됨. |
| **API 명세서의 형태** | 총 38개의 상세한 범용 REST API Endpoint 목록 나열 (Appendix 6.1) | 12개의 직관적인 **Next.js 내부 Server Action 및 Route Handlers 명세 표**로 교체됨 | 기존의 무거운 API 개발 설계론을 폐기하고, UI/Action 결합형 생태계에 맞는 명세로 변경됨. |
| **비기능 요구사항<br>제약 추가** | 컴포저블 SaaS 아키텍처에 의한 향후 모듈 교체(유연성)를 강조 | 문서 서두의 **제약사항(Constraints)**에 C-TEC 시리즈 8가지를 추가하여 기술 스택을 사전 고정 | 어떠한 경우에도 Next.js 모노레포와 Vercel 인프라 범위를 벗어나지 않도록 정책을 강제화함. |
