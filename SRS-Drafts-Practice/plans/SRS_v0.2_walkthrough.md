# SRS-001 바이브코딩 기반 아키텍처 개편 완료 보고서

PM님의 바이브코딩(AI 주도 코드 생성) 접근성을 극대화하고 개발 한계를 회피하기 위한 전략에 맞추어 `SRS-001_VPS_v1.0_Opus.md` 문서의 전체적인 개편을 완료했습니다. 변경된 주요 내용은 다음과 같습니다.

## 1. 제약사항(Constraints) 전면 갱신
* **반영 사항**: 문서 1.2.3 섹션에 제공해주신 기술 스택(C-TEC-001~007)을 명시적으로 추가했습니다.
* **효과**: AI에게 SRS 문서를 프롬프트로 제공할 시, AI가 기술 한계를 정확히 인지하고 Next.js 앱라우터 환경에 맞는 코드만 생성하도록 강제됩니다. 기존 레거시 설정(Redis, AWS 복수 결합 등)은 배제되었습니다.

## 2. 모놀리식 아키텍처 다이어그램 도입
* **반영 사항**: 3.5 컴포넌트 다이어그램, 6.2.3 클래스 다이어그램 전체 수정
* **개편 내용**: 기존의 `API Gateway`, 수많은 서버 노드(`Microservices`)로 나뉘었던 다이어그램들을 **Client(Browser) -> Server Action(Next.js) -> Supabase(Prisma) / Vercel AI SDK(Gemini)** 의 단일하고 직관적인 흐름으로 교체했습니다. 

## 3. 핵심 리스크 회피 시퀀스 적용
* **위험 1: 서버리스 타임아웃 방지 반영**
  * 3.6.1 및 6.3.1 시퀀스 다이어그램에서 통 엑셀 파일을 백엔드로 전송하던 방식을 **클라이언트(Browser)에서 SheetJS를 활용해 JSON으로 로컬 파싱 후 Server Action으로 전송**하는 구조로 전면 수정했습니다.
* **위험 2: 외부 연동 복잡도 다운사이징 반영** 
  * 높은 인증 보안 기준으로 인해 지속적인 버그를 양산할 가능성이 높은 LTI 1.3 표준(F7) 요구사항들을 최소화하여 "Phase 1에서는 Mock-up (모의 인증) 기능만 개발" 하도록 4.1.7 섹션을 다운사이징했습니다.

## 4. API 목록을 Server Actions로 대체
* **반영 사항**: 부록 6.1장의 `API Endpoint List` 섹션을 통째로 들어내고, `Server Actions & Route Handlers Master List` 로 변경했습니다.
* **개편 내용**: AI가 백엔드 REST API를 별도 설계하지 않고, `upload.ts`, `talent.ts` 등 Next.js의 Action 함수들을 설계하고 즉시 프론트엔드에서 연동시키도록 가이드합니다. 

### 💡 확인 방법 
`SRS-001_VPS_v1.0_Opus.md` 원본 파일에 진입해 Markdown/Mermaid 뷰어로 확인해 보시면, 기존과는 다르게 Next.js와 Vercel 인프라에 맞춰 깔끔하게 정돈된 아키텍처 흐름도를 보실 수 있습니다.
