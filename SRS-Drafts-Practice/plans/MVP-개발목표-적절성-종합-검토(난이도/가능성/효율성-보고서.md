# MVP 개발 목표 및 시스템/비용 효율성 종합 검토 보고서

**대상 문서:** `SRS_v0.2.md`
**작성 기준:** 시스템 아키텍처 및 비용 효율성(난이도, 실현 가능성, 효율성) 관점 통합 검토
**작성일:** 2026-04-21

---

## 1. 총평 (Executive Summary)

현재 설계된 `SRS_v0.2.md`의 아키텍처(Next.js App Router 모놀리식 구조, Vercel Serverless, Supabase, Vercel AI SDK)는 **"MVP의 빠른 시장 검증"이라는 비즈니스 목표에 매우 정확하게 부합하는 극한의 효율적 설계**입니다. 
복잡한 MSA(마이크로서비스) 구조와 불필요한 인프라(Redis, Python Backend 등)를 제거하고 "바이브코딩(AI 코드 생성) 최적화"에 초점을 맞춘 것은 **탁월한 기술적 의사결정**입니다. 

다만, 표준 연동(xAPI, Open Badge)에서의 오버엔지니어링 리스크와 AI 예측율 확보를 위한 프롬프트 엔지니어링 시간이 지연 요소로 작용할 수 있어 이에 대한 통제가 필요합니다.

---

## 2. 세부 관점별 검토 결과

### 2.1 개발 속도 관점 (MVP 단계의 과도한 시간 소요 회피)

**✅ 강점 (시간 절약 가속 요인)**
1. **Next.js 모놀리식 통합 및 Server Actions 활용**: 프론트엔드/백엔드 데이터 통신을 위한 별도의 API 명세서 작성 및 fetch 단위의 라우팅 코드를 생략할 수 있어 개발 생산성이 극적으로 향상됩니다.
2. **클라이언트단(Client-side) Excel 파싱 처리**: `SheetJS`를 통해 브라우저에서 엑셀을 파싱하여 가벼운 JSON 형태로만 서버리스(Server Action)에 넘기는 구조는, 서버로 거대한 바이너리 파일을 통째로 전송할 때 발생하는 복잡성(네트워크 지연, 파일 업로드 스트림 처리, 버퍼 오버플로우)을 사전에 차단합니다.
3. **LTI 1.3 통합 연기 (Mock-up) (REQ-FUNC-035)**: 보안 서명 체계가 매우 이질적이고 복잡한 LTI 연동을 MVP 단계에서 모의 테스트로만 격리시킨 것은 일정 지연(Debugging Hell)을 방지하는 훌륭한 결정입니다.
4. **UI 컴포넌트 표준화 (Tailwind + shadcn/ui)**: Cursor 등 AI 코딩 툴이 가장 잘 이해하는 스택이므로 뷰(View) 레벨의 개발 시간을 80% 이상 단축시킬 수 있습니다.

**⚠️ 리스크 및 보완 제안 (개발 지연 방지)**
* **AI 매핑 프롬프트 튜닝 (REQ-FUNC-002)**: AI가 100% 매핑을 맞추게 하려는 시도는 끝없는 프롬프트 튜닝 시간을 요구합니다. 초기 MVP 방점은 매핑 정확도를 극대화하는 것보다는, 화면(REQ-FUNC-003)에서 **"사용자가 쉽게 보정할 수 있는 유연하고 쾌적한 피드백 UI"**를 만드는 데 두어야 개발 속도가 확보됩니다.
* **복잡한 통계/대시보드 쿼리**: 초기부터 데이터 웨어하우스 수준의 복잡한 쿼리를 만들지 말고, Prisma를 이용해 간단히 데이터를 불러온 뒤 애플리케이션 레벨(Node.js)에서 처리하여 개발 기간을 줄일 것을 권장합니다.

### 2.2 외부 연동 목표와 기술 스택 관점 (복잡성 및 오픈소스 등 적절성)

**✅ 강점 (아키텍처 적절성)**
1. **서버리스 한계 우회 (Timeout 극복)**: Vercel의 단기 실행 제약(Hobby/Pro 환경에서의 수 초 내 Timeout)을 방어하기 위해 클라이언트 파싱 기반 구조(EXT-01)를 채택한 것은 플랫폼 제약을 잘 스터디한 안전한 선택입니다.
2. **AI 인프라 최소화 (Vercel AI SDK)**: 별도의 Python(FastAPI/Flask) 백엔드 서버를 띄워 유지보수 포인트를 늘리지 않고, Next.js 내부에서 언어모델의 프롬프트를 스트리밍 통제하는 것은 아키텍처 스택을 크게 경량화하여 안정성을 높입니다. 

**⚠️ 리스크 및 보완 제안 (연동 한계 및 복잡도 대응)**
* **표준 시스템 (xAPI, Open Badge) 초기 오버엔지니어링 주의**: (EXT-02, EXT-04) IMS 표준 스펙은 매우 방대합니다. MVP에서는 별도 SaaS 연결이나 거대한 자체 구축을 시도하지 말고, Supabase DB 내부에 표준 JSON 규격으로 적재만 해두고 **"데이터 발송 채비"**만 마치는 형태로 구현 범위를 제한해야 MVP 출시(M9 기준)를 맞출 수 있습니다.
* **벤더 종속 (Vendor Lock-in) 대응 준비**: Vercel의 Serverless 구조에 매우 강하게 결합됩니다. 향후 CSAP 인증(본사업 전환 등)을 위해 네이버클라우드, NHN클라우드로 마이그레이션이 필요해질 때를 대비하여 백엔드의 핵심 비즈니스 로직은 Next Component 내에 섞지 말고, `actions/` 폴더 내 독립된 모듈로 잘 격리(Separation of concerns)해 두어야 나중에 Dockerizing 하기가 쉽습니다.

### 2.3 운영 소요 비용 관점 (클라우드 컴퓨팅 기반 비용 산정)

초기 파일럿 검증(5~20개 기관 한정 투입) 수준에서의 인프라/API 고정 지출을 추산했을 때 **가장 비용 효율적인 스타트업 SaaS 비용 구조**를 가집니다.

* **예상되는 클라우드 운영 비용 구조 (MVP 단계, 월 기준)**
    1. **Frontend / Serverless Hosting (Vercel Pro)**: `$20 / 월`
        * 트래픽이 간헐적이고 예측 가능한 관리자 B2B 시스템이므로, 24시간 띄워두어야 하는 상시 대기 서버(EC2) 대신 서버리스 구조를 택하여 불필요한 Idle Cost(유휴 유지비)를 0원으로 수렴시킴.
    2. **Database & Storage (Supabase Pro)**: `$25 / 월`
        * 오픈소스 PostgreSQL 기반으로 확장성이 보장되며 초기 50GB 용량은 엑셀, JSON 등 텍스트 중심의 B2B 데이터 보관에 전혀 부족함 없음. PDF 리포트 등도 Supabase Storage에 보관.
    3. **LLM API 비용 (Google Gemini via AI SDK)**: `$10 ~ $30 / 월 예상`
        * Gemini 1.5 Flash 모델의 경우 입력 및 출력 토큰 비용이 극도로 낮음. 대용량 이미지 처리 없이, `엑셀 헤더 + 필드 샘플 데이터`와 JSON 스키마만을 텍스트로 처리하기 때문에 토큰 소모가 매우 효율적임.
    4. **(기타 선택) 도메인/이메일 SMTP 연동**: `$5 ~ $10 / 월` 

* **운영 비용 종합 결론**:
    * **월 $100 미만(약 13만 원 이하)**의 비용으로 엔터프라이즈급 AI 데이터 자동화 파이프라인의 MVP 파일럿 검증이 충분히 가능한 극한의 **"자본 효율형(Capital Efficient)" 인프라 구성**입니다. 클라우드 비용 리스크는 전혀 없다고 보아도 무방합니다.

## 3. 의사결정 및 Next Step 권고사항

1. **[실행] 우선순위 확정**: `SRS_v0.2.md` 문서는 기술/비용적으로 매우 훌륭하므로 별도 대규모 아키텍처 수정 없이 이대로 승인(Sign-off)하여 실 개발에 착수해도 무방합니다.
2. **[검토] 인증 구조 보완 여부 확인**: Supabase Auth와 PostgreSQL RLS(Row Level Security)를 통한 멀티테넌트 데이터 격리가 `SRS_v0.2` 요구사항을 충분히 충족하므로, 자체 세션 인증을 별도로 짜지 말고 **100% BaaS 기반 권한 분리**를 활용할 것을 개발 원칙에 명문화할 것을 권장합니다.
